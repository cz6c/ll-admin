# 前端 admin 开发

本章从页面开发者视角解释 `apps/admin` 的启动链路、路由机制、权限实现、状态管理与新增页面方式。

## 1. 前端的职责

`admin` 是一个 Vue 3 后台管理端，主要负责：

- 用户登录与退出
- 保存 token
- 调用后端接口
- 根据后端菜单动态生成路由
- 渲染系统管理与监控页面
- 提供后台常用 UI 交互模式

## 2. 目录结构说明

```text
apps/admin/
├─ src/
│  ├─ api/            接口封装
│  ├─ assets/         静态资源与样式
│  ├─ components/     全局组件
│  ├─ directives/     全局指令
│  ├─ hooks/          复用逻辑
│  ├─ layout/         布局系统
│  ├─ router/         路由与守卫
│  ├─ store/          Pinia 状态管理
│  ├─ utils/          请求、鉴权、工具函数
│  ├─ views/          页面
│  ├─ App.vue         应用根组件
│  └─ main.ts         启动入口
└─ package.json
```

## 3. 前端启动时做了什么

`src/main.ts` 是前端入口，主要会：

- 创建 Vue 应用
- 注册 Pinia
- 注册路由
- 注册路由守卫
- 注册全局组件
- 注册全局指令
- 引入全局样式

因此，如果你要理解整个前端是怎么跑起来的，优先看 `main.ts`。

## 4. 路由是怎么组织的

前端路由分为两部分：

### 4.1 静态路由

这些页面不依赖后端菜单：

- 登录页
- 首页
- 404
- 重定向页

### 4.2 动态路由

这些页面来源于后端返回的菜单树：

- 用户管理
- 角色管理
- 菜单管理
- 部门管理
- 参数管理
- 公告管理
- 日志监控等

## 5. 动态菜单与动态路由怎么工作

这是前端最关键的一块。

流程如下：

1. 登录成功后保存 token
2. 请求当前用户信息
3. 请求 `/getRouters`
4. 前端把菜单中的 `component` 字符串映射到 `views` 目录对应组件
5. 调用 `router.addRoute()` 动态注入路由
6. 左侧菜单与页面访问能力同时生效

这意味着：

- 菜单显示与否由后端控制
- 页面是否能访问也受后端菜单返回结果影响

## 6. 权限守卫怎么理解

路由守卫主要做三件事：

- 没有 token 时跳到登录页
- 有 token 但未拉取用户信息时，先拉用户信息与菜单
- 切换路由时处理进度条、标题、消息关闭等通用行为

## 7. 请求层怎么理解

项目使用自定义 Axios 封装，能力比较完整：

- 统一设置 `baseURL`
- 自动加 `Authorization`
- 防止重复提交
- 统一错误提示
- 处理 `401`
- 自动刷新 token
- 刷新成功后重放失败请求

如果接口联调异常，优先排查请求层配置而不是先怀疑页面组件。

## 8. 登录态怎么管理

登录态涉及三块：

- `utils/auth`：负责 token 的读写
- `store/modules/auth.ts`：负责登录、获取用户信息、登出
- `router/guard.ts`：负责页面访问控制

改登录时通常需要同时关注这三层。

## 9. 布局系统怎么理解

后台整体布局由 `layout` 目录负责，主要包括：

- 左侧菜单 Sidebar
- 顶部导航 Navbar
- 标签页 TagsView
- 内容区 AppMain
- 设置抽屉 Settings

如果你只是新增业务页面，通常不需要改布局层。

## 10. 页面开发的基本模式

当前项目大多数后台页面都遵循类似模式：

1. 在 `src/api` 中封装接口
2. 在 `src/views` 中新建页面目录
3. 列表页负责查询、表格、操作按钮
4. 弹窗组件负责新增与编辑
5. 通过菜单配置决定该页面是否出现在侧边栏

## 11. 如何新增一个前端页面

推荐步骤：

1. 在 `src/views` 下新建页面目录
2. 在 `src/api` 下新增对应接口封装
3. 如果需要表单弹窗，在页面目录的 `components` 下拆分编辑组件
4. 让后端返回该页面的菜单配置
5. 重新登录或刷新菜单缓存，验证是否出现在左侧菜单中

## 12. 如何接入新接口

建议按下面方式接入：

1. 先在 Swagger 确认接口路径与参数
2. 在 `src/api` 中写请求函数
3. 给页面组件引入该请求函数
4. 对表单与返回数据做类型约束
5. 联调时优先在浏览器网络面板核对请求体与响应体

## 13. 为什么前端要依赖后端菜单

因为当前项目是后端驱动前端权限的设计：

- 用户拥有哪些页面权限，不由前端硬编码
- 页面入口、图标、标题、组件路径都可由后端菜单数据驱动

优点是权限统一；缺点是联调时前后端菜单配置必须同步。

## 14. 常用开发关注点

前端开发最常碰到的问题通常有：

- 登录成功但菜单不显示
- 页面文件已写好但路由进不去
- 接口返回了但表格不渲染
- token 过期后页面不断重定向
- 菜单 `component` 字符串与视图文件路径不一致

## 15. Electron 说明

`admin` 除了浏览器模式，还内置 Electron 打包能力。对于一般后台页面开发，你可以先只把它理解为普通 Web 管理端；只有在需要桌面端发布时再关注 Electron 目录。

## 16. 页面开发建议

- 一个页面只负责一个业务主题
- 复杂弹窗拆到 `components/` 下
- API 封装和页面逻辑分离
- 列表页优先保持统一交互习惯
- 菜单字段、路由字段、视图文件路径三者要一致
